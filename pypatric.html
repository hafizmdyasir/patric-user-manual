Introduction Last updated: 2024-08-08

PyPaTriC stands for Python - Particle Tracker in C++. This python package has been designed to aid in the creation of the input scripts for the PaTriC charged particle tracker.

How it Works

After installing this package, the user can import the pypatric.deck_helper module, which imports all the pre-defined keywords, classes, and functions. This makes everything in the package available for use.

The most important use of PyPaTriC is to provide functions to setup the input script for PaTriC. This makes the process as simple as calling these functions with the required arguments. PyPaTriC performs a quick check on the argument and ensures that any unexpected error does not show up during program execution.

Package Structure

The PyPaTriC package has the following rough structure

	| pypatric
	| ---- keys.py
	| ---- classes.py
	| ---- units.py
	| ---- electric.py
	| ---- magnetic.py
	| ---- deck_helper.py

Installation

PyPaTriC is available with most python package repositories with the package name codecat_pypatric. This is done to ensure uniqueness of package name.

To install PyPaTriC, you can use pip or apt-get. The exact command will vary depending on your pacakge manager, but it typically looks like pip install codecat_pypatric.

Modules in PyPaTriC

The modules in pypatric are listed below. Their descriptions follow in the succeeding sections, which will help familiarize the user with the nuances of the program and allow them to bend PaTriC to their will.

keys.py

This module contains all the pre-defined constants that are used to setup the input script for PaTriC.

classes.py

This module defines various classes that can be used to set up the initial conditions for the target particles, as well as the electric and magnetic fields.

units.py

This module defines a few constants and methods that can help the user convert quantities across commonly used units.

electric.py

Classes that are exclusive to defining electric fields are defined in this module.

magnetic.py

Classes that are exclusive to defining magnetic fields are defined in this module.

deck_helper.py

This module brings together all the modules listed above and defines the functions that the user can call to define the input script. It is recommended that you import this module into your input script for everything to work perfectly.

keys.py

There isn't much notable about this module. It is typically used directly by the program itself or imported indirectly through other modules. Most end-users will have no need to go into the bowels of this module.

However, a few constants of interest are listed below:

  • BORIS: Integer constant used to signal the use of the Boris push for the integration of the equation of motion.
  • VAY: Integer constant used to signal the use of the Vay integrator.
  • HIGUERA_CARY: Integer constant used to signal the use of the Higuera-Cary integrator.

classes.py

A description of each of the classes defined here is important. These classes are used to define initial conditions and electric and magnetic fields.

The Random Class

This class is used to request random placement of particles. An instance of this class is passed inside a tuple to the initialPosition parameter of the Particle dictionary. The constructor for Random has the signature
Random((x_min, y_min, z_min), (x_max, y_max, z_max)).

When you pass an object of this type to the initialPosition parameter, the x-,y-, and z-components of the initial positions of the particles are randomly picked from the range [x_min, x_max], [y_min, y_max], and [z_min, z_max].

The Uniform Class

This class is used to request uniform values for the initial positions and/or velocities for the target particles. The constructor signature is similar as the Random class, i.e.,
Uniform((x_min, y_min, z_min), (x_max, y_max, z_max)).

Passing an object to this type to the initialPosition or initialVelocity parameter, the initial positions and/or velocities are uniformly distributed between the particles.

The Maxwellian Class

This class is exclusively for the initialVelocity parameter. It takes a single floating point parameter that corresponds to the temperature in Kelvin.

Its job is to instruct PaTriC to draw the initial velocities of the particles from a Maxwellian distribution centered at 0 with FWHM related to the temperature provided.

Tip

The Maxwellian class constructor takes its input in kelvins. If you wish to work in electron-volts, a helper function called eVToKelvin has been defined in the units module for your convenience.

Warning

In python, tuples are typically defined using a pair of parantheses (). However, when you request Uniform or Random or Maxwellian placement, there will be only a single item in the tuple. In such a case, the code (Maxwellian(temp)) will be considered the same as Maxwellian(temp), i.e., the tuple won't be created. To get around this, always place a guard empty element in the tuple.
Use (Maxwellian(temp), ).

The Static Class

This class is used to instruct PaTriC to set a specific value for the initial position, initial velocity, electric field, and/or the magnetic fields. When used for the initial positions or initial velocities, the number of elements in the tuple passed to the initialPosition or initialVelocity parameter must exactly match the number of target particles in the system. Each entry in the tuple will then correspond to an individual particle.

The constructor has the signature Static(x_component, y_component, z_component).

Tip

Suppose you want to simulate 10 particles with the same initial positions. In such a case, you don't need to create 10 separate objects of type Static. Instead, use the fact that python allows you to extend the length of a tuple by using the * operator. You can just set initialPosition as tuple([Static(x,y,z)])*10.

The Output Dict

The function of the Output dictionary is to inform PaTriC how to dump the data from the simulation. This includes the file name for output, the file path, any optional data you want to put in the info file, and the variables to dump.

The Output dictionary is created using the syntax createOutput(fileName: str, filePath: str, fileHeaders: Tuple[str, ...], dumpInfoFile: bool, dumpVariables: Tuple[str, ...]). A description is as follows:

File Name and Path

The fileName argument is a string representing the name of the output file. It should not include any extension. PaTriC will automatically assign .dat extension to the output files. The filePath argument is a string representing the directory path where the output file will be saved. If the directory does not exist, it will be created automatically.

fileHeaders Argument

This is a tuple of strings that will be written to an info file dumped alongside the output files, containing basic information about the simulation run. Note that regardless of the number of particles, only a single info file is created and it has the same name as provided via fileName.

dumpInfoFile Flag

If you want, you can disable the creation of the info file by passing False to this parameter. The default value is True and thus, info file will be created by default.

dumpVariables Argument

This is a tuple of strings that lists the variables you want to dump. Supported values inside this tuple are as follows:

  • POS_(X,Y,Z): The (x,y,z)-component of position.
  • VEL_(X,Y,Z): The (x,y,z)-component of velocity.
  • E_FIELD_(X,Y,Z): The (x,y,z)-component of the electric field calculated at each time instant.
  • B_FIELD_(X,Y,Z): The (x,y,z)-component of the magnetic field calculated at each time instant
  • GAMMA: The relativistic factor γ at each time instant.

Analyzing Output

The simulation data from PaTriC is saved as simple .dat files that are human readable and compatible with almost all plotting software. The plotters that ship with PaTriC are under construction and for now, your best bet is to use the numpy.loadtxt and use matplotlib to plot the data as per your convenience.

Data Structure

The output data file contains tabspace separated columns, with each column corresponding to a particular variable, and each row corresponding to a particular time-step. All data is dumped in SI units, in scientific format, with precision of 12 decimal places.

For each particle you load in the simulation, a separate output file will be created, and numbers starting with 0 will be appended at the end of the file name, corresponding to each particle.

The first row of is a header containing the description of each of the columns. The order of these columns is fixed and if a particular variable is not requested in the dumpVariables tuple in the Output dictionary, its column is not created in the output file.

Order of Columns in Output File
Column Variable Dumped Units
1 Time Always Seconds
2,3,4 Position (X,Y,Z) On request Metres
5,6,7 Velocity (X,Y,Z) On request Metres per second
8 Relativistic Factor γ On request Dimensionless
9,10,11 Electric Field (X,Y,Z) On request Volts per metre
12,13,14 Magnetic Field (X,Y,Z) On request Tesla

Note

Since the electric and magnetic fields are calculated at the beginning of each iteration, the very last iteration does not have information about them. Thus, the last row for electric and magnetic fields will contain NaN.

The Info File

This is a plain text file named as <fileName>.info. The strings you provide in the fileHeaders tuple will be written at the top of this info file, with each string in a new line.

The info file also contains the the total time taken (tt) to run the program, the time-step used (dt), and the number of tierations (ni). And finally, the target information including charge, mass, number of particles, and initial conditions for each of the particles are also dumped in this file. This can come in handy when analyzing the output.
